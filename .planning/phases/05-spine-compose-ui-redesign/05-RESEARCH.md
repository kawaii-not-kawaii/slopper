## Summary

Phase 5 is a pure visual re-skin: design tokens, nav component, two new composables (`SpineResumeCard`, `ChapterStrip`), and per-screen color/type updates. No new routes, no business logic changes. The codebase is in a clean state: `StashTheme` is applied exactly once — at `MainActivity.setContent` — which means the M3 `MaterialTheme` color scheme propagates automatically to all screens. Most screen composables use `MaterialTheme.colorScheme.*` for their colors, so the majority of token migration is concentrated in `Color.kt` → `Theme.kt` wiring. The exception is direct `StashColors.*` call sites (11 files, ~112 occurrences) which require explicit rename.

The `StashColors` object has 22 named tokens. `SpineColors` replaces them all with 19 tokens plus 2 new tokens (`AccentCool`, `AccentCoolDim`) and removes 5 old tokens (`SurfaceHighest`, `ScrimStrong`, `ScrimMedium`, `ScrimSoft`, `Divider`, `AccentSecondary`, `AccentSecondaryDim`). This is the highest-risk area: `AccentSecondary` is used 4 times in `PlayerControls.kt` (teal-themed loop/bookmark controls) and must map to `AccentCool`.

`StashTheme` wraps `StashExtraColors` via `LocalStashColors` — this entire mechanism is deleted in Spine. Composables that call `LocalStashColors.current` (6 hits: `SceneCard.kt`, `DetailScreen.kt`, `PlayerControls.kt`, `BrowseScreen.kt`) must be updated to use `SpineColors` directly.

**Primary recommendation:** Execute in strict wave order. Plan 5.1 must produce a green build before Plan 5.2/5.3 touch any screen file. The hard-delete migration means the codebase will not compile mid-migration — each Plan must be complete before commit.

`Theme.kt` currently defines:

## A3 — BottomNav.kt Current Structure

### What `MainBottomBar` Currently Does (verified, full file read)

`MainBottomBar` renders a standard M3 `NavigationBar` with:
- `containerColor = StashColors.SurfaceLow` (full-width anchored bar)
- `tonalElevation = 0.dp`
- Items from `visibleIds` list (default: Home, Scenes, Studios, Performers — 4 tabs)
- Always-appended "More" tab (`Icons.Filled.Tune`) that calls `onOpenMore`
- `NavigationBarItemDefaults.colors(...)` hardwired to `StashColors.AccentPrimary` for selected state

### What Must Be Replaced

The entire `NavigationBar { ... }` block is replaced with a custom `Row` composable inside a `Box` with `Modifier.align(Alignment.BottomCenter)`. Per D-10, this overlay Box lives at the root of `StashAppContent`, not as `Scaffold.bottomBar`.

**New pill structure (from D-10 + UI-SPEC):**
```
Box (root StashAppContent)
├── [content with Modifier.padding(bottom = 100.dp)]
└── Row pill bar (Modifier.align(BottomCenter), padding bottom 14dp, horizontal 12dp)
    ├── background: SpineColors.Surface.copy(alpha = 0.92f)
    ├── border: 1dp SpineColors.Border
    ├── shape: RoundedCornerShape(16.dp)
    ├── shadow: optional drop shadow
    ├── blur: API 31+ only (D-02)
    └── [up to 4 tab items + no More tab]
```

**Tab item states:**
- Inactive: padding 9dp/12dp, transparent bg, `SpineColors.OnSurfaceVariant`, icon 18dp, NO label
- Active: padding 9dp/12dp, `SpineColors.AccentPrimary` bg, `SpineColors.AccentOnPrimary`, icon 18dp + Space Grotesk 11sp W600 label

### What Must Be Kept

The `MainNavItem` data class, `MainNavItems` object, and `visibleIds` parameter logic are all kept — the nav item model is unchanged. Only the rendering composable changes.

### MoreSheet — Current Location and What Changes

`MoreSheet` composable is in `BottomNav.kt` (lines 169–236), not a separate file. [VERIFIED: full file read]

## Common Pitfalls

### Pitfall 1: Mid-Migration Compile Failure
**What goes wrong:** Deleting `StashColors` in `Color.kt` before updating all 11 call-site files causes a build failure that blocks the branch for the entire Plan 5.1 duration.
**Why it happens:** D-07 is a hard delete — no aliases. Any file still importing `StashColors` fails immediately.
**How to avoid:** The planner must order Plan 5.1 tasks so `Color.kt` is the LAST file updated (not first). Update all call sites → then rename Color.kt. Alternatively, rename `StashColors` to `SpineColors` as a class-level rename first (all tokens present), then update token names individually.
**Warning signs:** Any `Unresolved reference: StashColors` in build output.

### Pitfall 2: `LocalStashColors` Dangling References
**What goes wrong:** 6 composables call `val extra = LocalStashColors.current`. If `LocalStashColors` is deleted from Theme.kt without updating these composables, they fail to compile. The files are: `SceneCard.kt`, `DetailScreen.kt`, `PlayerControls.kt`, `BrowseScreen.kt`, `SettingsScreen.kt` (+ possibly others not caught by grep).
**Why it happens:** `LocalStashColors` is a `CompositionLocal` — it's not a class import that Kotlin will catch as "unused import" until the definition is removed.
**How to avoid:** Run the grep (`grep -rn 'LocalStashColors'`) before committing Theme.kt changes. Confirm 0 hits.

### Pitfall 3: `AccentSecondary` Has No Direct Spine Equivalent
**What goes wrong:** `StashColors.AccentSecondary` (teal #4FC8D9) maps to `SpineColors.AccentCool` (#7FB6FF, blue). It is used in `PlayerControls.kt` for loop/bookmark/progress-ring elements. These will change from teal to blue — which is correct per Spine — but if the mapping is missed, the code compiles against a deleted token.
**Why it happens:** The name change is not a rename — it's a semantic remapping (teal secondary → cool blue accent).
**How to avoid:** The migration map in A1 lists all 4 `AccentSecondary` hits. Treat them as a distinct named set.

### Pitfall 4: `StashExtraColors` / `LocalStashColors` Provider in `StashTheme`
**What goes wrong:** `StashTheme` currently calls `CompositionLocalProvider(LocalStashColors provides StashExtraColors())`. If this line is not removed when `LocalStashColors` is deleted, it creates a compile error in `Theme.kt` itself.
**How to avoid:** Plan 5.1 task (c) — Theme.kt update — explicitly removes the `CompositionLocalProvider` wrapper and the `LocalStashColors`/`StashExtraColors` definitions simultaneously.

### Pitfall 5: Pill Nav Bottom Padding Clips Content
**What goes wrong:** If the root content composable does not have `Modifier.padding(bottom = 100.dp)` (approx. pill height 56dp + 14dp gap + safe area), content in list screens will be clipped behind the pill on devices with long content.
**Why it happens:** The pill is an overlay Box, not `Scaffold.bottomBar`, so Scaffold's `contentWindowInsets` does not account for it.
**How to avoid:** D-10 specifies `padding(bottom = 100.dp)` at the root content level. This is applied in `StashAppContent`, not per-screen.

### Pitfall 6: `Font(GoogleFont(...), provider)` Import Collision
**What goes wrong:** `Type.kt` currently imports `androidx.compose.ui.text.font.FontFamily`, `Font`, etc. After adding the Google Fonts library, `Font` has two definitions: `androidx.compose.ui.text.font.Font` and `androidx.compose.ui.text.googlefonts.Font`. Using the wrong one at each call site causes a type mismatch.
**How to avoid:** Use fully-qualified import or alias: `import androidx.compose.ui.text.googlefonts.Font as GoogleFontLoader`. Call `FontFamily(GoogleFontLoader(GoogleFont("Space Grotesk"), provider, ...))`.

### Pitfall 7: Resume Bar Width Bug in SceneCard
**What goes wrong:** The current resume progress bar uses `Modifier.size(width = 0.dp, height = 3.dp)` for the outer Box — this causes the `Box(Modifier.fillMaxSize())` inside it to always fill 100% width regardless of `resumeFraction`. The bar always appears full.
**Why it happens:** Pre-existing bug — `size(width=0.dp)` + `fillMaxSize()` inside renders at 100% of the parent, not a fraction.
**How to avoid:** Replace with `Modifier.fillMaxWidth(resumeFraction).height(3.dp)` directly on the progress Box. Fix this during Spine SceneCard update.
