## PERF-08 Shuffle Root Cause

**Finding:** Listener accumulation was NOT the cause. `addListener` is called exactly once
(lazy init) and `removeListener` once (onCleared). No accumulation.

**Root cause confirmed:** `PlayerQueue.advance()` returns `null` when the shuffled queue
is exhausted with `RepeatMode.OFF` (the default). `onSceneEnded()` was calling
`queue.advance() ?: return` — the null case was handled silently with no user feedback.
The user perceived this as a "hang" because the last video ended and nothing visibly changed.

**Note:** `RepeatMode.ALL` wrap-around was already implemented correctly in
`PlayerQueue.advance()` (lines 52-57) — no change needed there.

**Fix applied:** `PlayerViewModel.onSceneEnded()` now emits an `"End of queue"` banner
when `advance()` returns null, using the existing `_state.update` + `clearBannerLater()`
pattern already used for shuffle/repeat/speed banners.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] CompilationMode.None() syntax fix**
- **Found during:** Task 2 (compile verification)
- **Issue:** `CompilationMode.None` treated as a companion object expression — the
  benchmark-macro 1.3.3 API defines `None` as an inner class, not a Kotlin `object`.
  Compiler error: "Classifier 'class None : CompilationMode' does not have a companion
  object, so it cannot be used as an expression."
- **Fix:** Changed `CompilationMode.None` to `CompilationMode.None()` in
  `ColdStartBenchmark.kt` line 67.
- **Files modified:** `baselineprofile/src/main/java/io/stashapp/android/baselineprofile/ColdStartBenchmark.kt`
